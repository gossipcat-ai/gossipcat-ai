# Session Memory — Orchestrator Continuity Across Sessions

**Date:** 2026-03-29
**Status:** Spec (alpha feature)

---

## The Problem

Every gossipcat session starts from zero. The orchestrator (the LLM that dispatches tasks, chooses agents, interprets results) has no memory of prior sessions. It doesn't know:

- What the team accomplished yesterday
- Which approaches failed and why
- Which agents are strong/weak at specific tasks
- What's in progress (specs, half-implemented features)
- User preferences expressed in prior sessions

This makes the orchestrator the dumbest part of a system that's getting smarter everywhere else. Agents have cognitive memory. Performance signals accumulate. Skills evolve. But the thing directing all of it starts fresh every time.

## The Thesis

**Gossipcat is the only multi-agent system where the orchestrator gets smarter across sessions.** CrewAI, AutoGen, LangGraph — none have session-level orchestrator memory. This is gossipcat's differentiator: **the mesh remembers.**

## Design

### Core Principle: The orchestrator is a virtual agent

Agent ID: `_project`. Uses the same memory infrastructure as real agents:

```
.gossip/agents/_project/memory/
├── MEMORY.md              ← Project memory index (loaded into bootstrap)
├── tasks.jsonl            ← Session entries (warmth-scored, compacted)
├── archive.jsonl          ← Cold sessions evicted by compactor
└── knowledge/
    ├── 2026-03-29T22-session.md   ← Cognitive session summary
    ├── 2026-03-28T19-consensus.md ← Key consensus insights
    └── ...                         ← Max 10, oldest pruned
```

**Why reuse agent infrastructure?** MemoryWriter, MemoryCompactor, AgentMemoryReader all work. Warmth-based decay works. Cognitive summaries work. Bootstrap generator already knows how to read agent memory. Zero new persistence code.

### What gets written and when

#### 1. After each consensus run (automatic)

When `runConsensus()` completes with findings, write a knowledge entry for `_project`:

```
What was reviewed, by which agents
How many confirmed/disputed/unverified
Key confirmed findings (top 3 by confidence)
Agent observations (who agreed with whom, who was disputed)
```

This is generated by the utility LLM — same `generateCognitiveSummary` pattern, different prompt.

**Trigger:** Inside `DispatchPipeline.runConsensus()`, after signal writing.
**Cost:** One utility LLM call per consensus run (~500 tokens).

#### 2. Session save (user-triggered MCP tool)

New tool: `gossip_session_save`

```typescript
gossip_session_save(notes?: string)
```

Generates a comprehensive session summary from:
- Session gossip entries (in-memory task summaries from this session)
- Consensus runs this session (from consensus-history or signals)
- Agent performance changes (scores at session start vs now)
- Skill index changes (new bindings, toggles)
- Git log since session start (what was committed)
- User notes (optional freeform context)

The LLM synthesizes all of this into a cognitive summary:

```markdown
---
name: Session 2026-03-29
description: 9 commits, consensus engine hardening, cognitive memory, skill index
importance: 0.9
lastAccessed: 2026-03-29
---

## What shipped
You shipped 9 commits this session: native consensus gap fix, per-agent
skill index, UNVERIFIED consensus tag, Memory Tier 3, cognitive summaries...

## What you learned
The STOP_WORDS approach for finding dedup failed after 3 iterations. Static
word lists can't distinguish boilerplate from signal. The final solution uses
raw Jaccard with high thresholds (0.6/0.7). Don't revisit this.

## Agent observations
Gemini agents consistently cite wrong line numbers in cross-review. Sonnet
reads actual code and cites correctly. The UNVERIFIED system handles this.

## In progress
Dashboard spec at docs/specs/2026-03-29-dashboard-design.md needs changes:
POST not GET for auth, consensus-history unbounded growth, path traversal
on memory endpoint...

## For next session
- Update dashboard spec with agent review findings
- Build Phase 1: Overview + Agents tabs
- Consider normalizeSkillName Windows path guard
```

**Trigger:** User calls `gossip_session_save()` before ending session.
**Also:** The tool prints a reminder: "Session saved. Next session, call gossip_bootstrap() to load context."

#### 3. Consensus knowledge (automatic, lightweight)

After each consensus run, write a compact entry (not the full report):

```markdown
Consensus run: 3 agents reviewed consensus-engine.ts
4 confirmed, 1 disputed, 7 unverified
Key: STOP_WORDS removal confirmed, UNVERIFIED tag validated
Sonnet found 3 findings gemini missed. Gemini cited wrong lines.
```

**Trigger:** Automatic after `runConsensus()`. No user action needed.
**Cost:** One utility LLM call.

### How it loads into the next session

`BootstrapGenerator.generate()` already reads agent memory. Add `_project` as a special case:

```typescript
// In renderTeamPrompt(), after agent summaries:
const projectMemory = this.readProjectMemory();
if (projectMemory) {
  prompt += '\n## Session Context\n\n' + projectMemory;
}
```

`readProjectMemory()` uses `AgentMemoryReader.loadMemory('_project', '')` to get the most relevant knowledge files by warmth. The top 2-3 entries become the session context section in the bootstrap.

The bootstrap prompt then looks like:

```markdown
# Gossipcat — Multi-Agent Orchestration

## Your Team
- sonnet-reviewer: anthropic/claude-sonnet-4-6 (reviewer) ...
- gemini-reviewer: google/gemini-2.5-pro (reviewer) ...

## Session Context

Last session (2026-03-29): 9 commits shipped. Key lessons:
- STOP_WORDS approach failed for dedup — use raw Jaccard instead
- UNVERIFIED consensus tag is the real fix for line-number disputes
- Cognitive memory summaries are the biggest quality improvement
- Gemini agents cite wrong line numbers — cross-reference with sonnet

In progress: Dashboard spec needs updates from 3-agent review...

## Tools
...
```

**Cap:** Session context section is capped at 2000 chars in the bootstrap. The full detail lives in the knowledge files for deeper reference.

### Compaction

Same warmth-based decay as agent memory:
- `maxEntries = 10` for task entries (session log)
- `MAX_KNOWLEDGE_FILES = 10` for knowledge files
- Warmth = `importance × (1 / (1 + days / 30))`
- After 30 days, old sessions fade. Critical lessons stay (high importance).

The dynamic cap from Tier 3 applies: sessions with many consensus runs get more retention.

### The prompt for session summary generation

This is the most critical piece. The prompt must produce actionable, specific, warning-oriented summaries:

```
You are writing a project memory entry that will be loaded into the
orchestrator's context at the start of the next session. This context
helps the orchestrator make better decisions about agent dispatch,
task planning, and avoiding past mistakes.

Write as a briefing for a new team lead taking over. Focus on:

1. WHAT SHIPPED — concrete deliverables, not process descriptions.
   Name the features, cite file paths if relevant.

2. WHAT FAILED AND WHY — approaches that didn't work. Be specific
   about why, so future sessions don't retry them.
   Format: "We tried X because Y. It failed because Z. The fix was W."

3. AGENT OBSERVATIONS — which agents are reliable for what, who
   hallucinates, who finds things others miss. Back with data.

4. IN PROGRESS — specs in review, half-built features, known bugs.
   Include file paths and what specifically needs to happen next.

5. USER PREFERENCES — how the user likes to work (e.g., "always
   run multi-agent review before merging", "prefers conservative
   thresholds over aggressive optimization").

Rules:
- Max 500 words
- No preamble. Start with "## What shipped"
- Cite file paths when referencing code or specs
- Include specific numbers (commit count, finding count, test count)
- Warnings > accomplishments — it's more useful to know what NOT to do
```

### MCP tool interface

```typescript
gossip_session_save(notes?: string)
// notes: optional freeform user context ("we're focusing on security hardening")
// Returns: the generated summary text
// Side effects:
//   1. Writes knowledge entry to .gossip/agents/_project/memory/knowledge/
//   2. Writes task entry to .gossip/agents/_project/memory/tasks.jsonl
//   3. Rebuilds .gossip/agents/_project/memory/MEMORY.md index
//   4. Prints reminder about gossip_bootstrap() for next session
```

### Data sources for summary generation

| Source | What it provides | How to access |
|--------|-----------------|---------------|
| Session gossip (in-memory) | Task summaries from this session | `pipeline.sessionGossip` |
| agent-performance.jsonl | Signal counts, score changes | `PerformanceReader.getScores()` |
| Skill index | Binding changes this session | Compare index mtime vs session start |
| Git log | Commits since session start | `git log --oneline --since="<boot time>"` |
| Consensus reports | Findings, agent attributions | From `runConsensus()` return values (cached in-memory) |
| User notes | Freeform context | Passed to the tool |

### What this replaces

The manual `.gossip/next-session.md` file (which the user wrote by hand after a previous session) becomes automated. The `gossip_session_save()` tool generates the same kind of content but richer, more consistent, and based on actual data rather than what the user remembers to write down.

## Implementation scope

| Component | Effort | New code |
|-----------|--------|----------|
| `_project` memory directory creation | Small | In MemoryWriter or bootstrap |
| Consensus knowledge auto-write | Small | 20 lines in `runConsensus()` |
| `gossip_session_save` MCP tool | Medium | New tool, LLM prompt, data gathering |
| Bootstrap reads `_project` memory | Small | 15 lines in BootstrapGenerator |
| Session start timestamp tracking | Small | Store boot time for git log range |

Total: ~150 lines of new code. Most complexity is in the LLM prompt, not the plumbing.

## Non-goals (v1)

- **Auto-detect session end** — MCP has no shutdown hook. User triggers manually.
- **Cross-project memory** — each project has its own `_project` agent. No sharing.
- **Memory editing UI** — the dashboard (future) could show session history, but not edit it.
- **Conflict resolution** — if two users work on the same project, last-write-wins for session save.

## Risk

**Low.** Reuses all existing memory infrastructure. The only new code paths are the MCP tool and the bootstrap integration. Failure mode: if the LLM produces a bad summary, it's one knowledge file that fades via warmth decay. No permanent damage.

**Privacy note:** Session summaries contain information about what was worked on (file paths, feature names, agent observations). This stays in `.gossip/` which is local and gitignored. No data leaves the machine unless the user explicitly shares it.
